# PRD (Product Requirements Document) 가이드

> 작성일: 2025-12-22
> 목적: 프로젝트 시작 전 문서화 체계 이해를 위한 참고 자료

---

## 1. PRD란?

**PRD (Product Requirements Document)** = 제품 요구사항 정의서

- **"무엇(What)을, 왜(Why) 만드는가?"** 를 정의하는 문서
- 개발 시작 전에 작성하는 가장 기초적인 기획 문서
- 프로젝트의 나침반 역할

---

## 2. PRD가 중요한 이유

| 이유 | 설명 |
|------|------|
| **명확한 목표 설정** | "무엇을 만들 것인가"를 명확히 정의, 개발 중 방향을 잃지 않게 해줌 |
| **시간과 비용 절약** | 코딩 전에 문제를 발견하면 수정 비용이 10배 이상 저렴 |
| **일관된 사용자 경험** | 전체 흐름을 미리 설계하여 UX 일관성 유지 |
| **소통 도구** | 팀원, 이해관계자와의 의사소통 기준 문서 |

---

## 3. 프로젝트 문서 체계

```
┌─────────────────────────────────────────────────────────────┐
│                    📋 PRD (제품 요구사항)                     │
│         "무엇(What)을, 왜(Why) 만드는가?"                     │
│  • 목표, 타겟 사용자, 핵심 기능, 성공 지표                      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│              📐 기술 설계 문서 (Technical Spec)               │
│           "어떻게(How) 만들 것인가?"                          │
│  • 스키마/데이터 구조 설계                                     │
│  • API 설계 (CRUD 라우터)                                     │
│  • 시스템 아키텍처                                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│              📝 구현 계획 (Implementation Plan)               │
│           "언제, 어떤 순서로 개발하는가?"                       │
│  • 개발 단계                                                  │
│  • 일정                                                       │
│  • 작업 순서                                                  │
└─────────────────────────────────────────────────────────────┘
```

---

## 4. 문서별 포함 내용 비교

| 문서 | 포함 내용 | 예시 |
|------|----------|------|
| **PRD** | 제품 비전, 사용자 스토리, 기능 목록 | "사용자는 상품을 장바구니에 담을 수 있다" |
| **기술 설계서** | 데이터 모델, API 명세, 아키텍처 | "Product 스키마: name, price, stock..." |
| **구현 계획** | 개발 단계, 일정, 작업 순서 | "1주차: 백엔드 스키마 → 2주차: API..." |

---

## 5. 추천 문서 작성 순서

| 순서 | 문서 | 내용 |
|------|------|------|
| 1 | **PRD** | 왜 만드는지, 누구를 위한 것인지, 핵심 기능은 무엇인지 |
| 2 | **와이어프레임/목업** | 화면 구성과 사용자 흐름 |
| 3 | **기술 스펙** | 기술 스택, DB 스키마, API 설계 |
| 4 | **구현 계획** | 개발 단계별 마일스톤 |

## 6. PRD 템플릿

```markdown
# [프로젝트명] PRD

## 1. 프로젝트 개요
- **프로젝트 이름**: 
- **목표 및 비전**: 
- **타겟 사용자**: 

## 2. 문제 정의
- **기존의 문제점**: 
- **해결하고자 하는 핵심 문제**: 

## 3. 핵심 기능 (MVP)
### 사용자 기능
- [ ] 회원가입 / 로그인
- [ ] 상품 검색 / 필터링
- [ ] 장바구니
- [ ] 주문 / 결제

### 관리자 기능
- [ ] 상품 관리 (CRUD)
- [ ] 주문 관리
- [ ] 회원 관리

## 4. 비기능 요구사항
- **성능**: 페이지 로딩 3초 이내
- **보안**: HTTPS, 결제 보안
- **반응형**: 모바일/태블릿/데스크톱 지원

## 5. 성공 지표 (KPI)
- 가입 전환율: 목표 X%
- 장바구니 이탈률: 목표 X% 이하
- 평균 주문 완료 시간: X분 이내

## 6. 타임라인
- **Phase 1 (MVP)**: 2주 - 핵심 기능
- **Phase 2**: 2주 - 고급 기능
- **Phase 3**: 1주 - 테스트 및 배포
```

## 7. 개발 5단계 (구현 계획에 속함)

PRD 완성 후, 실제 개발은 다음 순서로 진행:

```
1️⃣ 스키마 데이터 구조 만들기 (백엔드)
    └─ MongoDB/PostgreSQL 모델 정의

2️⃣ 서버에서 CRUD 라우터 만들기 (백엔드)
    └─ Express/NestJS 라우터 및 컨트롤러

3️⃣ 프론트엔드 페이지 디자인
    └─ 컴포넌트 및 페이지 UI 구현

4️⃣ 서버와 프론트 연결 로직
    └─ API 호출, 상태 관리 연동

5️⃣ 테스트
    └─ 단위 테스트, 통합 테스트, E2E 테스트
```

---

## 8. 실무 흐름 요약

```
1️⃣ PRD 작성 → "장바구니 기능이 필요해"
      ↓
2️⃣ 기술 설계 → "Cart 스키마, /api/cart 엔드포인트 필요"
      ↓
3️⃣ 구현 계획 → "1. 스키마 → 2. API → 3. UI → 4. 연결 → 5. 테스트"
      ↓
4️⃣ 실제 개발
      ↓
5️⃣ 테스트 & 배포
```

## 9. 프로젝트 폴더 구조 예시

```
📁 프로젝트
├── 📁 docs/
│   ├── 📄 PRD.md                    ← "무엇을 만들지" 정의
│   ├── 📄 technical_spec.md          ← 기술 설계 문서
│   ├── 📄 implementation_plan.md     ← 구현 계획
│   └── 📄 design_system.md          ← UI/UX 가이드
├── 📁 client/                        ← 프론트엔드
├── 📁 server/                        ← 백엔드
└── 📄 README.md
```
